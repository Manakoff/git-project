# 🛠️ Практический репозиторий: Освоение Git & GitHub

Добро пожаловать! Этот репозиторий создан в учебных и практических целях для полной отработки навыков работы с системой контроля версий **Git** и платформой **GitHub**. Здесь сымитирован реальный рабочий процесс (Workflow) в команде.

---

## 🚀 Ключевые компетенции
* **Уровень:** Уверенный hard-skills по Git (Console / CLI).
* **Командная работа:** Полное понимание процессов ветвления, код-ревью и работы с Pull Requests.
* **Культура кода:** Написание понятных, атомарных и структурированных коммитов.

---

## 📚 Что было изучено и отработано на практике

### 🔹 1. Базовые операции и фиксация изменений
* Проверка состояния репозитория и отслеживание файлов (`git status`, `git diff`).
* Подготовка и индексация изменений (`git add`).
* Создание правильных, атомарных коммитов (`git commit`).
* Работа с историей проекта (`git log`, `git reflog`).

### 🔹 2. Ветвление и слияние (Branching & Merging)
* Изучение различных **стратегий и моделей ветвления** (Feature Branch Workflow, основы Git Flow).
* Создание, переключение и навигация между ветками (`git branch`, `git checkout`, `git switch`).
* Безопасное удаление локальных и удаленных веток (`git branch -d`, `git push origin --delete`).
* Слияние веток (`git merge`), разрешение конфликтов слияния и работа с **Pull Requests** на GitHub.

### 🔹 3. Отмена изменений и работа с историей (Undo & Reset)
* Откаты релизов и изменений на разных этапах работы (`git reset --soft / --hard`).
* Отмена коммитов с созданием обратного коммита (`git revert`).
* Удаление ошибочных коммитов, изменение и редактирование последнего коммита (`git commit --amend`).

### 🔹 4. Культура коммитов (Good Commits)
* Изучены правила хорошего тона при написании Git-сообщений (понятные заголовки, краткое описание сути изменений).
* Использование императивного наклонения и атомарный подход (один коммит — одна изолированная задача).

---

## 🛠️ Пример рабочего процесса в репозитории

На примере коммитов в данном репозитории (таких как `Merge pull request #1 from Manakoff/feature`) была успешно отработана следующая схема:
1. Создание изолированной ветки `feature` из ветки `main`.
2. Внесение изменений в исходный код (`src/`) и создание коммитов.
3. Отправка ветки на удаленный сервер (`git push`).
4. Открытие Pull Request, проведение проверки и финальное слияние (Merge) в основную ветку.
